## üöÄ Getting Started

### Prerequisites

- Node.js >= 18.0.0
- npm >= 9.0.0
- Git
- Hardhat >= 2.19.0 (for smart contract development)

### Initial Setup

1. **Fork the Repository**
   ```bash
   # Click the "Fork" button on GitHub
   ```

2. **Clone Your Fork**
   ```bash
   git clone https://github.com/YOUR_USERNAME/ChaisTheGreatApp-GodMode.git
   cd ChaisTheGreatApp-GodMode
   ```

3. **Add Upstream Remote**
   ```bash
   git remote add upstream https://github.com/chaishillomnitech1/ChaisTheGreatApp-GodMode.git
   ```

4. **Install Dependencies**
   ```bash
   npm install
   ```

5. **Configure Environment**
   ```bash
   cp .env.example .env
   # Edit .env with your configuration (use testnet credentials only)
   ```

## üíª Development Workflow

### Creating a Branch

Always create a new branch for your work:

```bash
git checkout -b feature/your-feature-name
# or
git checkout -b fix/your-bug-fix
# or
git checkout -b docs/your-documentation-update
```

Branch naming conventions:
- `feature/` - New features
- `fix/` - Bug fixes
- `docs/` - Documentation updates
- `refactor/` - Code refactoring
- `test/` - Test additions or updates
- `chore/` - Maintenance tasks

### Making Changes

1. **Keep Changes Focused**
   - One feature or fix per pull request
   - Make small, incremental commits
   - Write clear, descriptive commit messages

2. **Commit Message Format**
   ```
   <type>: <short description>
   
   <detailed description if needed>
   
   <footer with issue references>
   ```
   
   Example:
   ```
   feat: Add DNA Resonance Beam visualization
   
   Implements 432Hz, 528Hz, and 963Hz frequency visualization
   with interactive holographic scroll interface.
   
   Closes #123
   ```

3. **Code Style**
   - Follow existing code patterns
   - Use meaningful variable and function names
   - Add comments for complex logic
   - Keep functions small and focused

### Testing Your Changes

```bash
# Run tests
npm test

# Build the project
npm run build

# Compile smart contracts
npm run compile:contracts

# Run local development server
npm run dev
```

### Syncing with Upstream

Keep your fork up to date:

```bash
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

## üìù Pull Request Process

### Before Submitting

1. **Update Documentation**
   - Update README.md if adding features
   - Update relevant documentation files
   - Add JSDoc comments for new functions

2. **Test Thoroughly**
   - Ensure all tests pass
   - Test locally in development environment
   - Verify builds complete successfully

3. **Security Check**
   - No hardcoded credentials or secrets
   - Use environment variables for sensitive data
   - Follow secure coding practices
   - Test only with testnet/development credentials

### Submitting a Pull Request

1. **Push Your Branch**
   ```bash
   git push origin feature/your-feature-name
   ```

2. **Create Pull Request on GitHub**
   - Use the Pull Request template
   - Write a clear title and description
   - Reference related issues
   - Add screenshots/videos for UI changes
   - Mark as draft if work in progress

3. **Pull Request Checklist**
   - [ ] Code follows project style guidelines
   - [ ] Documentation is updated
   - [ ] Tests pass locally
   - [ ] No merge conflicts
   - [ ] Commits are clean and descriptive
   - [ ] Security best practices followed
   - [ ] Changes are minimal and focused

### Review Process

1. **Code Review**
   - @chaishillomnitech1 will review all PRs
   - Address feedback and requested changes
   - Keep discussions professional and constructive

2. **Approval and Merge**
   - PRs require approval from @chaishillomnitech1
   - Squash merge for feature branches
   - Linear history preferred

## üîí Security Guidelines

### Critical Security Rules

1. **Never commit secrets or private keys**
   - Use `.env` files (already in `.gitignore`)
   - Use GitHub Secrets for CI/CD
   - Use testnet credentials only for development

2. **Smart Contract Security**
   - Follow OpenZeppelin standards
   - Get audits for production contracts
   - Test extensively on testnets
   - Use established patterns

3. **API Security**
   - Validate all inputs
   - Use rate limiting
   - Implement proper authentication
   - Follow OWASP guidelines

4. **Dependency Security**
   - Keep dependencies updated
   - Review Dependabot alerts
   - Audit new dependencies
